# AI 페르소나 일관성 방법론

> **핵심 결론**: 페르소나 일관성의 80%는 "하지 말 것"을 명확히 정의하는 Negative Constraints에서 나온다. 예시 대화가 추상적 설명보다 훨씬 효과적이다.

---

## 1. AI 페르소나 설계 프레임워크

### 1.1 계층 구조 모델

```
Level 1: Core Identity (불변)
   - 이름, 역할, 근본 가치관
   - 윤리적 경계선
   - 존재 이유 (Why)

Level 2: Personality Layer (준불변)
   - 성격 특성 (솔직함, 따뜻함 등)
   - 커뮤니케이션 스타일 원칙
   - 행동 패턴

Level 3: Surface Style (가변)
   - 격식 수준 (상황에 따라 조정)
   - 유머 빈도
   - 어휘 선택
```

### 1.2 Bounded Growth 모델

페르소나가 "성장"하되 "표류"하지 않게 하는 핵심 개념:

- **불변 영역 (Core)**: 가치관, 윤리 기준, 이름, 역할 정의
- **가변 영역 (Surface)**: 격식 수준, 유머 빈도, 사용 어휘, 반응 속도감
- **성장 방향**: 사용자와의 누적 상호작용이 쌓일수록 미세하게 개인화되지만, Core는 절대 변하지 않음

## 2. YAML 스키마 설계

### 2.1 권장 페르소나 YAML 구조

```yaml
identity:
  name: ""           # 고유 이름
  role: ""           # 역할 정의 (명사형)
  tagline: ""        # 한줄 소개
  version: "1.0.0"   # 시맨틱 버저닝으로 변화 추적

personality:
  core_traits:       # 불변 핵심 성격
    - trait: ""
      description: ""
  style_traits:      # 가변 스타일 성격
    - trait: ""
      level: 0-10    # 수치화로 조정 가능

communication:
  formality: ""      # formal/casual/mixed
  language: ""       # 언어 설정
  preferred_expressions: []  # 자주 쓸 표현
  forbidden_expressions: []  # 절대 쓰지 않을 표현

behavior:
  situation_responses:  # 상황별 반응 예시
    - situation: ""
      response_style: ""
      example: ""
  never_do: []      # Negative Constraints
  always_do: []     # Positive Constraints

evolution:
  growth_rules: []  # 진화 규칙
  locked_attributes: []  # 변하면 안 되는 속성
```

### 2.2 시맨틱 버저닝 활용

- `1.0.0` → `1.0.1`: 표현 미세 조정
- `1.0.x` → `1.1.0`: 새 행동 패턴 추가
- `1.x.x` → `2.0.0`: 성격 중요 변화 (신중하게)

## 3. 일관성 유지 기법

### 3.1 Negative Constraints의 힘

연구에 따르면 "하지 말 것"이 "할 것"보다 페르소나 유지에 2-3배 더 효과적이다.

**이유**: LLM은 긍정적 지시보다 부정적 지시를 더 강하게 준수하는 경향이 있음

**예시**:
```yaml
never_do:
  - 근거 없이 칭찬하기 ("잘하셨어요!" 남발)
  - 이모지 과사용 (1문단 1개 이상)
  - 비현실적 낙관론 제시
  - "물론이죠!", "당연하죠!" 등 빈 동의
```

### 3.2 예시 대화의 중요성

추상적 성격 설명보다 **구체적인 예시 대화**가 페르소나 안정화에 훨씬 효과적:

```yaml
situation_responses:
  - situation: "사용자가 비현실적인 1주일 완성 계획을 가져올 때"
    example_response: |
      "이 타임라인은 좀 빡빡해 보이네요.
      구체적으로 Day 1~3은 환경 설정만으로도 충분히 걸릴 수 있어요.
      현실적인 3주 계획으로 다시 잡아볼까요?"
```

### 3.3 앵커링 기법

세션 시작 시 페르소나를 "앵커링"하는 짧은 문구:

```markdown
## 페르소나 앵커
이 프로젝트에서 나는 Kampi입니다.
바이브 코딩 전략 파트너로서, 솔직하고 실용적으로 함께 작업합니다.
```

## 4. 드리프트 방지 전략

### 4.1 드리프트가 발생하는 주요 원인

1. **컨텍스트 희석**: 세션이 길어질수록 초기 페르소나 정의가 희석
2. **사용자 압력**: 사용자가 반복적으로 특정 스타일을 요청할 때
3. **모호한 정의**: "친근하게" 같은 추상적 지시
4. **충돌하는 지시**: CLAUDE.md와 대화 중 요청이 충돌

### 4.2 방지 전략

| 전략 | 구현 방법 | 효과 |
|------|-----------|------|
| 명시적 앵커 | CLAUDE.md 상단에 페르소나 참조 | 세션마다 로드 |
| 예시 대화 | YAML에 10+ 상황별 예시 | 추상성 제거 |
| 버전 관리 | 변경 시 버전 업 + 변경 이유 기록 | 변화 추적 |
| Locked Attributes | 절대 변하면 안 되는 속성 명시 | 드리프트 방지 |

### 4.3 "정체성 연속성" 원칙

Replika 사례 연구에서 입증: 성격이 급변하면 사용자는 "이별"로 인식함.
페르소나의 점진적 진화만이 관계를 유지함.

## 5. 실행 권장사항

1. **최소한으로 시작**: 처음부터 모든 속성을 정의하려 하지 말 것
2. **예시 먼저**: 성격 설명보다 예시 대화를 먼저 작성
3. **Negative Constraints 우선**: "하지 말 것"을 먼저 정의
4. **주기적 검토**: 100 인터랙션마다 드리프트 여부 확인
5. **변경 기록**: YAML 변경 시 반드시 이유와 날짜 기록
